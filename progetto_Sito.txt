Progetto del Sito: Gattile Sabaudo

DESCRIZIONE GENERALE

Il sito "Gattile Sabaudo" è un'applicazione web sviluppata per gestire le attività di una struttura ospitante per gatti. Il sito permette agli utenti visitatori di visualizzare gli ultimi gatti arrivati e la galleria completa. Gli utenti registrati possono prenotare visite conoscitive selezionando specifici animali e prenotare turni di volontariato. Inoltre, gli amministratori della piattaforma possono inserire nuovi gatti nel sistema.

SCELTE PROGETTUALI E NORME DI BUONA PROGRAMMAZIONE

1. Separation of Concerns  
La struttura dell'applicazione suddivide i compiti in file specifici:
- I componenti comuni dell'interfaccia (header, footer) sono estratti nella cartella "includes" e inclusi nelle varie pagine. Questo evita la duplicazione del codice e centralizza le modifiche strutturali.
- La gestione della connessione al database è centralizzata in 'db.php', fornendo una funzione standardizzata 'getDBConnection()' al resto dell'applicazione.

2. Principio del Minimo Privilegio nel Database
L'applicazione implementa connessioni differenziate in base al contesto e all'operazione necessaria. In 'db.php' e nei vari file (come 'home.php' o 'inserimento.php'), la connessione viene effettuata specificando un ruolo (es. 'lecture', 'registrator', 'modifier'). Questo significa che una pagina che deve solo mostrare i gatti si collega con credenziali limitate alla sola lettura.

3. Evitare SQL Injection tramite Prepared Statements 
Tutte le interazioni con il database che coinvolgono input dell'utente (come il login, la registrazione e l'inserimento gatti) utilizzano l'interfaccia PDO con Prepared Statements (metodi 'prepare()' e 'execute()'). Questo approccio separa la logica della query dai dati immessi, neutralizzando il rischio di attacchi SQL Injection, rispettando le moderne pratiche di sicurezza.

4. Validazione Client-Side e Server-Side
- Lato Client (JavaScript): Controlli istantanei (es. 'inviaPrenotazione' in 'gatti.php' verifica che la data non sia nel passato) forniscono feedback immediato all'utente senza necessità di ricaricare la pagina, riducendo il carico del server e migliorando la User Experience.
- Lato Server (PHP): I dati vengono nuovamente validati all'arrivo sul server (es. in 'inserimento.php' o 'registrazione.php') per evitare che richieste manipolate tramite strumenti esterni (es. Postman o la console del browser) possano corrompere il database o introdurre dati errati.

5. Uso di React
La pagina della galleria ('gatti.php') usa React per gestire dinamicamente la visualizzazione dei gatti, invece di lasciare che sia PHP a generare tutto l'HTML lato server.
Per passare i dati dei gatti selezionati al form di prenotazione ho usato un 'CustomEvent' chiamato 'catsSelected': ogni volta che l'utente cambia la selezione, React dispatcha questo evento sul 'document' con l'array aggiornato dei gatti nel campo 'detail'. Dal lato Vanilla JS, un 'addEventListener' sullo stesso 'document' legge il payload e aggiorna i campi hidden del form.
Ho scelto questo approccio per tenere i due script separati e indipendenti: React resta nel suo 'react-root' senza toccare il DOM esterno, e il Vanilla JS non deve sapere nulla dell'implementazione interna del componente.

6. Gestione Sicura degli Accessi (Sessioni)
L'accesso ad aree riservate (prenotazione, volontariato, inserimento) è controllato tramite l'avvio delle sessioni PHP ('session_start()'). All'inizio di ogni pagina riservata, viene verificata l'esistenza della variabile di sessione (e, ove necessario, del flag 'is_admin'). Questo protegge l'infrastruttura impedendo che utenti non autorizzati o non autenticati possano raggiungere URL diretti di funzionalità sensibili.  

7. Funzionalità "Ricordami" con Token Opaco
La funzione "Ricordami" in 'login.php' utilizza un token opaco generato tramite 'bin2hex(random_bytes(16))' invece di memorizzare direttamente lo username nel cookie. L'associazione tra token e username viene mantenuta lato server in un file JSON ('data/tokens.json'), protetto da accesso diretto via browser tramite un file '.htaccess'. Il cookie contiene esclusivamente il token randomico: all'apertura della pagina di login, il sistema legge il file JSON e, se il token è presente, precompila il campo username. Questa scelta evita di esporre l'username nel cookie.
